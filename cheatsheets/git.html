초기화 & 설정
git init
현재 디렉토리에 .git 폴더를 생성하여 버전 관리를 시작합니다.
새 Git 저장소 초기화
git clone <url>
원격 저장소의 전체 히스토리를 포함한 로컬 복사본을 생성합니다.
원격 저장소 복제
git config --global user.name "name"
사용자 이름 설정
git config --global user.email "email"
이메일 설정
기본 워크플로
git status
수정된 파일, 스테이징된 파일, 추적되지 않는 파일을 보여줍니다.
작업 디렉토리 상태 확인
git add <file>
특정 파일의 변경사항을 다음 커밋에 포함시킬 준비를 합니다.
파일을 스테이징 영역에 추가
git add .
현재 디렉토리 하위의 모든 변경된/새로운 파일을 스테이징합니다. .gitignore에 명시된 파일은 제외됩니다.
모든 변경사항 스테이징
git add -p
파일 내 변경사항을 hunk 단위로 나눠 선택적으로 스테이징할 수 있습니다. 하나의 파일에서 여러 논리적 변경을 분리할 때 유용합니다.
변경사항 선택적 스테이징
git commit -m "msg"
스테이징된 변경사항을 메시지와 함께 저장소 히스토리에 기록합니다.
커밋 생성
git commit --amend
마지막 커밋의 메시지나 내용을 수정합니다. 이미 push한 커밋은 수정하지 않는 것이 좋습니다.
마지막 커밋 수정 DANGER
git diff
스테이징되지 않은 변경사항을 diff 형식으로 보여줍니다.
워킹 디렉토리 변경사항 확인
git diff --staged
git add로 스테이징한 변경사항만 diff로 보여줍니다. --cached와 동일합니다.
스테이징된 변경사항 확인
브랜치 관리
git branch
로컬 브랜치 목록을 보여줍니다. 현재 브랜치에 * 표시가 됩니다.
브랜치 목록 조회
git branch -a
로컬과 원격의 모든 브랜치를 보여줍니다.
모든 브랜치 조회 (원격 포함)
git branch <name>
현재 커밋에서 새 브랜치를 생성하지만 전환하지는 않습니다.
새 브랜치 생성
git branch -d <name>
병합 완료된 브랜치만 삭제합니다. 미병합 브랜치는 -D 옵션을 사용해야 합니다.
브랜치 삭제 DANGER
git branch -D <name>
병합 여부와 관계없이 브랜치를 삭제합니다. 복구가 어려우므로 주의하세요.
브랜치 강제 삭제 DANGER
git branch -m <old> <new>
브랜치 이름 변경
브랜치 전환
git checkout <branch>
작업 디렉토리를 해당 브랜치의 상태로 변경합니다.
브랜치 전환
git checkout -b <name>
새 브랜치를 만들고 바로 전환합니다. git branch + git checkout을 한 번에 실행합니다.
브랜치 생성 + 전환
git switch <branch>
checkout의 브랜치 전환 기능만 분리한 명령어입니다. 더 명확한 의도를 표현합니다.
브랜치 전환 (Git 2.23+)
git switch -c <name>
브랜치 생성 + 전환 (Git 2.23+)
병합
git merge <branch>
지정한 브랜치의 변경사항을 현재 브랜치에 병합합니다. Fast-forward 또는 3-way merge가 수행됩니다.
브랜치 병합
git merge --no-ff <branch>
Fast-forward가 가능해도 병합 커밋을 생성합니다. 브랜치 히스토리를 보존할 때 유용합니다.
항상 병합 커밋 생성
git merge --abort
충돌이 발생한 병합을 취소하고 병합 전 상태로 돌아갑니다.
병합 충돌 시 병합 취소
git mergetool
충돌 해결 도구 실행
원격 저장소
git remote -v
등록된 원격 저장소의 이름과 URL을 보여줍니다.
원격 저장소 목록 확인
git remote add <name> <url>
원격 저장소 추가
git remote remove <name>
원격 저장소 제거
git remote rename <old> <new>
원격 저장소 이름 변경
git remote set-url <name> <url>
원격 URL 변경
동기화
git fetch
원격 저장소의 변경사항을 다운로드하지만 로컬 브랜치에 병합하지는 않습니다.
원격 변경사항 가져오기
git fetch --all
모든 원격에서 가져오기
git fetch --prune
원격에서 삭제된 브랜치의 로컬 추적 참조를 제거합니다.
삭제된 원격 브랜치 정리
git pull
원격의 변경사항을 가져와 현재 브랜치에 자동으로 병합합니다.
fetch + merge
git pull --rebase
병합 대신 rebase로 원격 변경사항을 적용합니다. 불필요한 병합 커밋을 방지하여 히스토리가 깔끔해집니다.
fetch + rebase
git push
로컬 커밋을 원격 저장소에 업로드합니다.
원격에 변경사항 전송
git push -u origin <branch>
-u 플래그로 추적 관계를 설정하면 이후 git push만으로 push할 수 있습니다.
업스트림 설정 + 푸시
git push --force-with-lease
다른 사람의 변경사항을 덮어쓰지 않도록 검증 후 강제 푸시합니다. --force보다 안전합니다.
안전한 강제 푸시 DANGER
git push --force
원격 히스토리를 로컬로 덮어씁니다. 다른 사람의 작업을 잃을 수 있으므로 매우 주의해야 합니다.
강제 푸시 DANGER
스테이징 취소
git restore <file>
파일을 마지막 커밋 상태로 되돌립니다. 수정한 내용이 사라지므로 주의하세요.
워킹 디렉토리 변경 취소 (Git 2.23+) DANGER
git restore --staged <file>
git add로 스테이징한 파일을 언스테이징합니다. 파일 내용은 변경되지 않습니다.
스테이징 취소
git checkout -- <file>
restore 명령어 이전 방식. 워킹 디렉토리의 변경을 취소합니다.
파일 변경 취소 (레거시)
git reset HEAD <file>
restore --staged 이전 방식. 스테이징 영역에서 파일을 제거합니다.
스테이징 취소 (레거시)
커밋 되돌리기
git revert <commit>
지정한 커밋의 변경을 취소하는 새로운 커밋을 생성합니다. 히스토리를 보존하므로 안전합니다.
커밋 되돌리기 (새 커밋 생성)
git reset --soft HEAD~1
커밋만 취소하고 변경사항은 스테이징 상태로 유지합니다.
마지막 커밋 취소 (변경사항 유지)
git reset --mixed HEAD~1
기본 옵션. 커밋을 취소하고 변경사항을 워킹 디렉토리에 유지합니다.
마지막 커밋 취소 (언스테이징)
git reset --hard HEAD~1
커밋과 모든 변경사항을 완전히 삭제합니다. 복구하려면 reflog을 사용해야 합니다.
마지막 커밋 완전 삭제 DANGER
git reset --hard <commit>
지정한 커밋 이후의 모든 변경사항이 사라집니다. 매우 위험한 명령어입니다.
특정 커밋으로 강제 리셋 DANGER
복구
git reflog
HEAD가 가리켰던 모든 커밋의 기록을 보여줍니다. reset --hard로 잃어버린 커밋도 찾을 수 있습니다.
참조 로그 확인
git checkout <reflog-hash>
reflog에서 복구
히스토리 조회
git log
커밋 히스토리 보기
git log --oneline
각 커밋을 해시 + 메시지 한 줄로 간결하게 표시합니다.
한 줄로 보기
git log --graph
브랜치와 병합 히스토리를 ASCII 그래프로 시각화합니다.
그래프로 보기
git log --oneline --graph --all
모든 브랜치의 히스토리를 그래프로 한 눈에 볼 수 있습니다.
전체 브랜치 그래프
git log -p
각 커밋의 diff를 함께 표시합니다.
변경 내용과 함께 보기
git log --stat
파일별 변경 통계 보기
git log --author="name"
특정 작성자 커밋만 보기
git log --since="2 weeks ago"
기간별 필터링
git log <file>
특정 파일의 히스토리
비교 & 검색
git diff <branch1>..<branch2>
브랜치 간 비교
git diff <commit1>..<commit2>
커밋 간 비교
git log -S "keyword"
특정 문자열이 추가되거나 제거된 커밋을 찾습니다. 코드 변경 추적에 매우 유용합니다.
코드 변경 내용 검색
git log --grep="keyword"
커밋 메시지 검색
git blame <file>
파일의 각 줄을 누가 언제 마지막으로 수정했는지 보여줍니다.
파일의 각 줄 작성자 확인
git show <commit>
특정 커밋 상세 보기
git shortlog -sn
작성자별 커밋 수 통계
임시 저장
git stash
현재 작업 중인 변경사항을 스택에 임시 저장하고 워킹 디렉토리를 깨끗한 상태로 만듭니다.
변경사항 임시 저장
git stash push -m "message"
설명 메시지를 붙여 stash를 구분하기 쉽게 합니다.
메시지와 함께 저장
git stash -u
untracked 파일도 함께 stash합니다.
추적되지 않는 파일 포함
git stash -p
변경사항을 hunk 단위로 선택하여 일부만 stash할 수 있습니다.
선택적으로 stash
복원 & 관리
git stash list
stash 목록 보기
git stash pop
가장 최근의 stash를 적용하고 stash 목록에서 제거합니다.
최근 stash 적용 + 삭제
git stash apply
stash를 적용하지만 목록에서 제거하지 않습니다. 여러 브랜치에 같은 변경을 적용할 때 유용합니다.
stash 적용 (삭제 안 함)
git stash apply stash@{n}
특정 stash 적용
git stash drop stash@{n}
특정 stash 삭제
git stash clear
모든 stash 삭제 DANGER
git stash show -p
stash 내용 확인
태그
git tag
태그 목록 보기
git tag <name>
단순히 특정 커밋을 가리키는 포인터입니다.
경량 태그 생성
git tag -a <name> -m "msg"
태그 작성자, 날짜, 메시지를 포함하는 완전한 태그 객체를 생성합니다. 릴리즈에 권장됩니다.
주석 태그 생성
git push --tags
모든 태그 푸시
git tag -d <name>
태그 삭제
리베이스
git rebase <branch>
현재 브랜치의 커밋들을 대상 브랜치 위에 재적용합니다. 깔끔한 선형 히스토리를 만들 수 있습니다.
리베이스
git rebase -i HEAD~n
최근 n개의 커밋을 편집할 수 있습니다. squash, reword, edit, drop 등의 작업이 가능합니다.
인터랙티브 리베이스
git rebase --onto <new> <old> <branch>
브랜치의 기반을 다른 커밋으로 변경합니다. 복잡한 브랜치 구조를 정리할 때 사용합니다.
리베이스 대상 변경
git rebase --continue
충돌 해결 후 계속
git rebase --abort
리베이스 취소
git rebase --skip
현재 커밋 건너뛰기
체리픽
git cherry-pick <commit>
다른 브랜치의 특정 커밋만 현재 브랜치에 적용합니다.
특정 커밋 가져오기
git cherry-pick <c1>..<c2>
지정한 범위의 커밋들을 순서대로 적용합니다. c1은 포함되지 않습니다.
범위 체리픽
git cherry-pick --no-commit <commit>
변경사항을 스테이징 상태로 가져옵니다. 여러 커밋을 하나로 합칠 때 유용합니다.
커밋 없이 변경만 적용
리베이스 주의: 이미 push한 커밋은 리베이스하지 마세요. 다른 사람의 히스토리가 꼬일 수 있습니다.
워크트리
git worktree add <path> <branch>
같은 저장소에서 별도의 작업 디렉토리를 생성합니다. 여러 브랜치를 동시에 작업할 때 유용합니다.
워크트리 추가
git worktree add -b <new-branch> <path>
새 브랜치로 워크트리 추가
git worktree list
워크트리 목록 보기
git worktree remove <path>
워크트리 제거
git worktree prune
유효하지 않은 워크트리 정리
서브모듈
git submodule add <url>
다른 Git 저장소를 현재 저장소의 하위 디렉토리로 포함합니다.
서브모듈 추가
git submodule init
서브모듈 초기화
git submodule update
서브모듈 업데이트
git submodule update --init --recursive
전체 서브모듈 재귀 업데이트
git clone --recurse-submodules <url>
서브모듈 포함 클론
정리
git clean -n
-n은 dry-run입니다. 실제로 삭제하지 않고 어떤 파일이 삭제될지 보여줍니다.
삭제될 파일 미리보기
git clean -fd
추적되지 않는 파일/디렉토리 삭제 DANGER
git gc
불필요한 파일을 정리하고 저장소를 압축합니다.
저장소 최적화
git fsck
저장소 무결성 검사
유용한 설정
git config --global core.editor "vim"
기본 편집기 설정
git config --global init.defaultBranch main
기본 브랜치명 설정
git config --global pull.rebase true
pull 시 rebase 사용
git config --global rerere.enabled true
같은 충돌을 반복해서 해결할 필요 없이 이전에 해결한 방법을 자동으로 적용합니다.
충돌 해결 기억
git config --global alias.co checkout
자주 사용하는 명령어에 짧은 별칭을 만들 수 있습니다.
별칭 설정
git config --list
설정 목록 보기
유용한 별칭 예시
git config --global alias.st status
st = status
git config --global alias.co checkout
co = checkout
git config --global alias.br branch
br = branch
git config --global alias.ci commit
ci = commit
git config --global alias.lg "log --oneline --graph --all"
lg = 그래프 로그
.gitignore
.gitignore 파일에 추적하지 않을 파일 패턴을 작성합니다. node_modules/, .env, *.log, .DS_Store 등이 일반적입니다.
git rm --cached <file>
이미 추적 중인 파일을 .gitignore에 추가한 후 이 명령으로 추적을 중단합니다. 파일 자체는 삭제되지 않습니다.
추적 중인 파일 무시 시작
git check-ignore -v <file>
